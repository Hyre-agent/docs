# Benchmarking SERV Reasoning on real DeFi decisions

**HYRE Engineering · June 11, 2026**

HYRE recently moved its seven decision endpoints — token verdicts, cross-chain
migration calls, LP strategy, `/ask` — to [SERV Reasoning](https://docs.openserv.ai/serv-reasoning/introduction)
by OpenServ. Before shipping, we benchmarked it against our previous stack on
the task that actually pays our bills: **turning live market data into a
financial decision a user (or an agent) acts on.**

The headline: SERV was the only configuration that combined **100% arithmetic
accuracy, zero costly-direction errors, and zero calls over our production
latency budget**. But the more interesting findings are in the details — including
one configuration that scored *higher* on the rubric and still lost the
production slot, and a truncation bug in our own stack that the benchmark
caught.

---

## What HYRE decision endpoints do, and what we benchmarked

Most HYRE endpoints return enriched listings — new tokens, pool APRs, TVL.
Getting those slightly wrong costs nothing. The seven **decision endpoints**
are different: their output is a signal someone executes. `migrate` means "pay
a bridge fee and move your position." A wrong `migrate` costs real money; a
wrong `stay` costs only opportunity.

We benchmarked the decision that is hardest to fake: **cross-chain yield
migration**, where the correct answer is derivable from arithmetic:

```
break_even_days = bridge_cost / (position × apy_gap / 365)
```

Our production prompt defines the signal bands: short break-even → `migrate`,
medium → `wait`, long (or no real gain) → `stay`. Because the ground truth is
computable, there is no judgment call in scoring.

## The configurations we compared

All three ran the **identical production system prompt and identical input
data**, JSON mode on, temperature 0.3:

| | Configuration |
|---|---|
| **SERV** | `gemini-flash-latest` via `inference-api.openserv.ai`, `reasoning_effort: low`, max_tokens 2500 |
| **Flash** | Gemini 2.5 Flash direct (Google API), thinking enabled, 4096-token output budget |
| **Flash-Lite** | Gemini 2.5 Flash-Lite direct — our previous default for these endpoints |

## Results

45 generated cases (15 per signal class), 135 calls total, single session:

| Metric | SERV | Flash | Flash-Lite |
|---|---|---|---|
| Arithmetic accuracy (break-even within ±15%) | **100%** | **100%** | 10% |
| Signal accuracy (rubric, answered) | 82.5% | **100%** | 62.2% |
| **Costly-direction errors** (said `migrate` wrongly) | **0** | **0** | 3 |
| Calls over 12s production budget | **0** | 18 (40%) | 0 |
| Malformed JSON | 5 (11%) | 0 | 0 |
| Latency p50 / p95 | 6.1s / 9.3s | 11.4s / 15.9s | 1.3s / 1.9s |
| Avg reasoning tokens | 760 | 2,026 | 0 |
| Avg cost per call | $0.0045 | $0.0006 | $0.0001 |

## The metric that matters

HYRE is a synchronous, paywalled API. A user pays per call and waits for the
answer; our cascade aborts any model that exceeds its timeout. That budget —
**12 seconds for the reasoning tier** — is as much a part of the spec as
accuracy.

Read the table through that lens:

- **Flash-Lite is fast and cheap and gets the arithmetic wrong 90% of the
  time.** It states break-even numbers confidently ("21 days" when the answer
  is 57) and made 3 costly-direction calls — recommending users pay bridge fees
  they shouldn't. This is our *previous* configuration for these endpoints.
  The benchmark retired it.
- **Flash scored a perfect rubric — and 40% of its calls would have failed in
  production.** It earns that score by thinking for ~2,000 tokens per call,
  pushing p50 latency to 11.4s against a 12s budget. A model that's right after
  the timeout is, operationally, a model that didn't answer.
- **SERV is the only configuration that fits the box:** perfect arithmetic,
  zero costly-direction errors, p95 well under budget, at $0.0045/call —
  7.5× Flash's per-call cost and still a rounding error against what the
  endpoint charges.

### Direction of error

SERV's rubric misses (7 of 40 answered) were **all in the conservative
direction** — saying `stay` on cases the rubric scores `migrate` or `wait`.
Notably, its arithmetic was correct *even on the cases it missed*: one case
computed break-even at exactly 8.7 days (correct) and still recommended
holding. It is cautious about spending the user's money, not wrong about the
math. For a financial API, that is the failure mode we would choose.

Flash-Lite's errors had no such structure: wrong arithmetic, and errors in
both directions, including the expensive one.

## Issues found, reported, and guarded

A benchmark that only produces a winner isn't doing its job. Three issues
surfaced:

1. **SERV emits malformed JSON ~11% of the time** even with
   `response_format: json_object` set (reported to the OpenServ team). Our
   adapter validates JSON on receipt and falls through to the Gemini cascade,
   so a malformed response costs latency, never correctness.
2. **SERV's no-parameter default is slow.** Without an explicit
   `reasoning_effort`, calls run ~9s — past most synchronous budgets. We pin
   `low` and a 12s timeout.
3. **Our own Gemini fallback had a truncation bug.** Flash's thinking tokens
   share the output budget; at our previous 2048-token cap, decision-grade
   prompts truncated mid-JSON. The benchmark smoke run caught it; the cap is
   now 4096. Benchmarks find bugs on both sides of the fence.

## What's live

SERV Reasoning is the primary model on all seven HYRE decision endpoints,
with `reasoning_effort: low`, a 12s timeout, JSON validation, and automatic
fallback through Gemini Flash → Flash-Lite → OpenRouter. Listings stay on the
fast tier — reasoning is spent only where the output is a decision.

Every HYRE response carries `model_used` (e.g.
`serv/google/gemini-3.5-flash`), so the live answer-rate is publicly
observable, one call at a time.

## Methodology

- **45 cases, 15 per signal class** (`migrate` / `wait` / `stay`), generated
  from a seeded PRNG for reproducibility. Positions $2k–$40k, APYs 4–8%,
  reputable destination protocols (Aave V3, Compound V3) so protocol
  reputation never confounds the signal.
- **Ground-truth bands sampled away from the rubric's decision edges**
  (migrate: 2–13 days; wait: 35–55; stay: 70–120 or negative gain), so no case
  is ambiguous under the production prompt's own rules.
- `break_even_days` and `net_apy` were **withheld from the input** — each model
  must do the arithmetic, exactly as our production endpoint expects when
  upstream data is partial.
- Arithmetic accuracy = the break-even figure stated in the model's narrative
  is within ±15% of truth. Signal accuracy = exact match with the rubric class.
- Same week, same prompts, same data, concurrency 4. Single-session run; small
  samples carry a few points of stochastic noise. Raw per-call results (JSON)
  and the generator script are available on request.
